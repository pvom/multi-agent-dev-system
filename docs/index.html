multi-agent-dev-system

A role-based methodology for shipping software with AI agents. Coding agents are fast but undisciplined — they refactor what you didn’t ask for, skip verification, and lose the thread. This is the operating model I put around them: specialized agents with isolated context, strict separation of duties, and Spec-Driven Development.

The pipeline

Request→ Lead — decompose→ Arq — specs→ Coder ‖→ QA gate→ Security gate→ Ops — commit & deploy

The roles

AgentOwnsNever does
LeadDecompose into SDD tasks, write PRs, coordinateImplement · commit
ArqStandards, specs, interfacesImplement · commit
Coder ‖Implement one atomic task; parallelizableDecide architecture · commit · scope-creep
QARun the suite, evidence-based verdictFix code · commit
SecuritySecrets/PII/brand audit — hard gate—
OpsThe only role that commits, merges, deploysImplement · decide architecture

The guardrails

  • Separation of duties — no agent both authors a change and ships it.
  • QA approves on evidence — a number, a screenshot, a file:line — never “looks fine”.
  • Security is a hard gate — 0 secrets, 0 PII, 0 client brands before anything outward-facing ships.
  • Human approval for irreversible/outward-facing actions (public repo, push, deploy).
  • Spec-Driven Development — every task atomic, with a verification criterion, closed by an atomic commit.

The QA role made real is a separate runnable repo: agentic-qa-gate — a Playwright merge gate with a runtime design-system audit.

Sanitized methodology write-up · applied to a production healthcare SaaS (MedTrack) · source on GitHub.
